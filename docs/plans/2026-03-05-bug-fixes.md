## Task 1: C1 — `useCheckRoutine`: bucle infinito por supabase sin memoizar

**Archivo:** `lib/hooks/useCheckRoutine.ts`

### ¿Por qué es un bug?

`createClient()` en la línea 27 se ejecuta en cada render y devuelve un nuevo objeto. Ese objeto está en el array de dependencias del `useEffect` (línea 74). React compara dependencias por referencia: nuevo objeto → efecto se re-ejecuta → `setChecking(true)` → re-render → nuevo objeto → bucle infinito.

El guard `if (hasRoutine !== null) return` aborta la query de DB pero el efecto sigue corriendo en cada render, ejecutando código innecesariamente y disparando actualizaciones de estado.

### ¿Qué ganamos?

- El hook se ejecuta **una sola vez** cuando `userId` está disponible, no en loop.
- Cualquier página que use `useCheckRoutine` deja de tener renders excesivos.
- Comportamiento idéntico al ya-correcto `useAuth.ts`.

### Cambios

**Paso 1:** En `lib/hooks/useCheckRoutine.ts`, cambiar el import:
```ts
// Antes:
import { useState, useEffect } from 'react'

// Después:
import { useState, useEffect, useMemo } from 'react'
```

**Paso 2:** Memoizar el cliente (línea 27):
```ts
// Antes:
const supabase = createClient()

// Después:
const supabase = useMemo(() => createClient(), [])
```

**Paso 3:** Limpiar el array de dependencias del `useEffect` (línea 74). `supabase` y `router` son ahora referencias estables y no necesitan estar:
```ts
// Antes:
}, [userId, skipCheck, hasRoutine, redirectOnFound, router, supabase])

// Después:
}, [userId, skipCheck, hasRoutine, redirectOnFound]) // eslint-disable-line react-hooks/exhaustive-deps
```

**Paso 4:** Eliminar los `console.log` de debug que van a producción (líneas 35, 53, 58, 62):
```ts
// Eliminar estas líneas:
console.log('🔍 useCheckRoutine: Verificando si existe rutina...')
console.log('✅ useCheckRoutine: Rutina encontrada')
console.log('🔄 useCheckRoutine: Redirigiendo a /rutinas...')
console.log('📝 useCheckRoutine: No hay rutina')
```

**Paso 5:** Verificar build:
```bash
npm run build
```
Esperado: sin errores.

**Paso 6:** Commit:
```bash
git add lib/hooks/useCheckRoutine.ts
git commit -m "fix: memoizar supabase en useCheckRoutine para evitar bucle infinito de renders"
```

## Task 3: C4 — `handleBlur`: side-effect dentro de setState updater + eslint suprimido

**Archivo:** `app/entrenar/[sesionId]/page.tsx` (líneas 73–91)

### ¿Por qué es un bug?

El código actual:
```ts
const handleBlur = useCallback((serieId: string, ejId: string) => {
  setSesion(prev => {
    // ...encuentra la serie...
    if (serie) {
      Promise.resolve().then(() => {     // ← side-effect dentro del updater
        actualizarSerie(supabase, {
          sesionId,                      // ← capturado del closure externo, no del updater
          ...
        })
      })
    }
    return prev
  })
}, [supabase]) // eslint-disable-line     ← oculta que sesionId falta como dep
```

**Problema 1:** En React 18 Strict Mode (activo en desarrollo), los updaters de `setState` se ejecutan **dos veces** para detectar side-effects. El `Promise.resolve().then(...)` dentro del updater se invoca dos veces → `actualizarSerie` se llama dos veces por cada blur → doble guardado en DB.

**Problema 2:** `sesionId` se captura del closure de `useCallback` pero NO está en el array de dependencias. Si la página alguna vez recibe un `sesionId` diferente (routing dinámico), `handleBlur` usaría el ID viejo.

**Problema 3:** El comentario `// eslint-disable-line` oculta que el dep array está incompleto.

### ¿Qué ganamos?

- Un solo guardado por blur, siempre con el `sesionId` correcto.
- Código más legible: lectura de estado y side-effect separados, sin updater abusado.
- Strict Mode en dev ya no causa dobles guardados.

### Cambios

Reemplazar `handleBlur` completo:

```ts
// Antes:
const handleBlur = useCallback((serieId: string, ejId: string) => {
  setSesion(prev => {
    if (!prev) return prev
    const ej = prev.ejercicios.find(e => e.rutina_ejercicio_id === ejId)
    const serie = ej?.series.find(s => s.id === serieId)
    if (serie) {
      Promise.resolve().then(() => {
        actualizarSerie(supabase, {
          id: serieId,
          sesionId,
          peso_kg: serie.peso_kg ? parseFloat(serie.peso_kg) : null,
          repeticiones: serie.repeticiones ? parseInt(serie.repeticiones) : null,
        })
      })
    }
    return prev
  })
}, [supabase]) // eslint-disable-line react-hooks/exhaustive-deps

// Después:
const handleBlur = useCallback((serieId: string, ejId: string) => {
  setSesion(prev => {
    if (!prev) return prev
    // Leer estado actual sin side-effects (safe para doble invocación de Strict Mode)
    const ej = prev.ejercicios.find(e => e.rutina_ejercicio_id === ejId)
    const serie = ej?.series.find(s => s.id === serieId)
    if (serie) {
      // Side-effect fuera del updater, con las dependencias explícitas correctas
      actualizarSerie(supabase, {
        id: serieId,
        sesionId,
        peso_kg: serie.peso_kg ? parseFloat(serie.peso_kg) : null,
        repeticiones: serie.repeticiones ? parseInt(serie.repeticiones) : null,
      })
    }
    return prev
  })
}, [supabase, sesionId])
```

## Task 4: I6 — `userId!` sin guard completo en páginas de sesión

**Archivos:**
- `app/entrenar/[sesionId]/page.tsx` (línea 35)
- `app/entrenamiento/[sesionId]/page.tsx` (línea ~37)

### ¿Por qué es un bug?

El código actual:
```ts
if (!authenticated) {
  router.replace('/')
  return
}

obtenerSesionActiva(supabase, sesionId, userId!)  // ← userId! sin verificar
```

`useAuth` puede retornar `authenticated: true` con `userId: null` por un instante durante la transición de estado (el evento de auth llega, `setAuthenticated(true)` se llama antes de que `user` se propague). En React 18 los state updates son batched, pero hay casos edge donde el render intermedio tiene `authenticated=true, userId=null`.

Cuando `userId` es `null`, `userId!` lo convierte a `null` casteado como `string`. `isValidUUID(null)` en el servicio retorna `false` → la función retorna `{ success: false }` → la página queda en estado "Sesión no encontrada" sin posibilidad de retry.

### ¿Qué ganamos?

- La página no puede quedar atrapada en un estado de error por una transición de auth de milisegundos.
- Eliminamos el non-null assertion operator que oculta el riesgo.

### Cambios

**En `app/entrenar/[sesionId]/page.tsx`**, cambiar el guard del useEffect:
```ts
// Antes:
if (!authenticated) {
  router.replace('/')
  return
}

obtenerSesionActiva(supabase, sesionId, userId!).then(...)

// Después:
if (!authenticated || !userId) {
  router.replace('/')
  return
}

obtenerSesionActiva(supabase, sesionId, userId).then(...)
```

También hacer lo mismo en `handleFinalizar` (línea 131):
```ts
// Antes:
const resultado = await finalizarSesion(supabase, sesionId, userId!)

// Después:
if (!userId) return
const resultado = await finalizarSesion(supabase, sesionId, userId)
```

**En `app/entrenamiento/[sesionId]/page.tsx`**, aplicar el mismo cambio en el guard del useEffect.

## Task 5: C3 — `eliminarSesion`: borra series sin verificar ownership

**Archivo:** `lib/services/sesion-service.ts` (líneas 494–510)

### ¿Por qué es un bug?

```ts
// Paso 1: borra las series solo por sesion_id (sin userId)
await supabase.from('sesion_series').delete().eq('sesion_id', sesionId)

// Paso 2: intenta borrar la sesión (con userId)
await supabase.from('sesiones').delete().eq('id', sesionId).eq('user_id', userId)
```

Si RLS no está habilitado en `sesion_series` (la tabla no aparece en el checklist de RLS del CLAUDE.md), un atacante que conozca el UUID de una sesión ajena puede llamar a esta función y borrar todas las series de esa sesión, aunque la sesión en sí esté protegida por `userId`.

El vector de ataque: el cliente llama directamente a `eliminarSesion` con el UUID de una sesión ajena (cliente JavaScript modificado) → el delete de `sesion_series` pasa → el delete de `sesiones` falla porque `user_id` no coincide → el resultado devuelve `success: true` porque... no, espera: el delete de sesiones falla → devuelve `{ success: false }`. Pero los datos de series ya fueron borrados. **Data loss parcial irrecuperable.**

### ¿Qué ganamos?

- Las series solo se borran si la sesión pertenece al usuario que hace el request.
- Operación atómica: si la sesión no es del usuario, no se toca nada.

### Cambios

Agregar verificación de ownership antes de borrar series:

```ts
export async function eliminarSesion(
  supabase: SupabaseClient,
  sesionId: string,
  userId: string
): Promise<ResultadoOperacion<null>> {
  if (!isValidUUID(sesionId) || !isValidUUID(userId)) {
    return { success: false, error: 'IDs inválidos' }
  }

  try {
    // Verificar ownership antes de tocar datos relacionados
    const { data: sesion, error: ownerError } = await supabase
      .from('sesiones')
      .select('id')
      .eq('id', sesionId)
      .eq('user_id', userId)
      .maybeSingle()

    if (ownerError || !sesion) {
      return { success: false, error: 'Sesión no encontrada' }
    }

    // Ownership confirmado — eliminar series primero (evita FK violation si no hay CASCADE)
    const { error: seriesError } = await supabase
      .from('sesion_series')
      .delete()
      .eq('sesion_id', sesionId)

    if (seriesError) {
      console.error('Error eliminando series de sesión:', seriesError)
      return { success: false, error: 'Error al eliminar el entrenamiento' }
    }

    const { error } = await supabase
      .from('sesiones')
      .delete()
      .eq('id', sesionId)
      .eq('user_id', userId)

    if (error) {
      console.error('Error eliminando sesión:', error)
      return { success: false, error: 'Error al eliminar el entrenamiento' }
    }

    return { success: true, data: null }
  } catch (err) {
    console.error('Error en eliminarSesion:', err)
    return { success: false, error: 'Error al eliminar el entrenamiento' }
  }
}
```

## Task 6: I7 — Rate limit se saltea silenciosamente cuando Supabase da error

**Archivo:** `app/api/generar-rutina/route.ts` (líneas 91–102)

### ¿Por qué es un bug?

```ts
const { count: rutinasRecientes, error: rateLimitError } = await supabase
  .from('rutinas')
  .select('id', { count: 'exact', head: true })
  .eq('user_id', userId)
  .gte('created_at', unMinutoAtras)

if (!rateLimitError && (rutinasRecientes ?? 0) >= 2) {  // ← si hay error, se salta el check
  return NextResponse.json({ error: '...' }, { status: 429 })
}
```

Si la query de conteo falla (DB saturada, RLS mal configurado, timeout), `rateLimitError` es truthy → `!rateLimitError` es `false` → el rate limit **no se aplica**. Un usuario que genere errores de DB intencionalmente (o que tenga mala suerte) puede llamar a la API ilimitadamente, generando llamadas a OpenAI sin restricción y costos sin techo.

### ¿Qué ganamos?

- El rate limit es **fail-closed**: cuando hay duda, se bloquea en lugar de dejar pasar.
- El error se loguea para diagnóstico pero no expone el problema al usuario.
- Costo de OpenAI acotado incluso ante fallos de DB.

### Cambios

```ts
// Antes:
if (!rateLimitError && (rutinasRecientes ?? 0) >= 2) {
  return NextResponse.json(
    { error: 'Has alcanzado el límite de generación. Espera un minuto antes de intentar de nuevo.' },
    { status: 429 }
  )
}

// Después:
if (rateLimitError) {
  console.error('[generar-rutina] Error en rate limit check:', rateLimitError)
}
if ((rutinasRecientes ?? 0) >= 2) {
  return NextResponse.json(
    { error: 'Has alcanzado el límite de generación. Espera un minuto antes de intentar de nuevo.' },
    { status: 429 }
  )
}
```

**Nota:** Si `rateLimitError` ocurre, `rutinasRecientes` será `null` → `null ?? 0` = `0` → el check `0 >= 2` es `false` → la request pasa. Esto sigue siendo fail-open cuando DB falla. Para ser realmente fail-closed, bloquear también en caso de error:

```ts
if (rateLimitError) {
  console.error('[generar-rutina] Error en rate limit check, bloqueando por precaución:', rateLimitError)
  return NextResponse.json(
    { error: 'Error temporal. Intentá de nuevo en un momento.' },
    { status: 503 }
  )
}
if ((rutinasRecientes ?? 0) >= 2) {
  return NextResponse.json(
    { error: 'Has alcanzado el límite de generación. Espera un minuto antes de intentar de nuevo.' },
    { status: 429 }
  )
}
```
